# カジノゲーム・コレクション

Python + PySide6 (Qt) 製のマルチプレイ対応カジノゲーム集。
LAN / インターネット越しに複数人で遊べる9種類のゲームと、
全ゲームを一括ホストするヘッドレス専用サーバを含む。

通信はすべて **HTTP/1.1 + JSON**。専用サーバは 80 番 (または
リバースプロキシ経由の 443/https) で公開でき、大学・会社など
ファイアウォールのあるネットワークの中からでも接続できる。

**クライアント (`client/`) とサーバ (`server/`) は独立**しており、
共有するのは純Pythonのゲームエンジン (`shared/`) のみ。
サーバは PySide6 に依存せず標準ライブラリだけで動くため、
Docker Compose で軽量にデプロイできる。

## 必要環境

- クライアント: Python 3.10+ / PySide6 (`pip install -r client/requirements.txt`)
- サーバ: Python 3.10+ のみ (依存パッケージなし)。または Docker

## 設定 (.env)

IPアドレス・ポート・サーバ名などは **`.env`** から取得する。
まず `sample.env` をコピーして編集する:

```bash
cp sample.env .env
```

| キー | 用途 | 既定値 |
|---|---|---|
| `SERVER_NAME` | サーバ名 (クライアントの接続設定と一致が必要) | GameServer |
| `SERVER_BIND` | サーバの待受アドレス (直接起動時) | 0.0.0.0 |
| `SERVER_PORT` | サーバの待受・公開ポート | 80 |
| `CLIENT_ADDRESS` | 接続ダイアログの初期アドレス (IP / ドメイン / URL) | 127.0.0.1 |
| `CLIENT_PORT` | サーバー接続モードの初期ポート | 80 |
| `CLIENT_SERVER_NAME` / `CLIENT_ROOM` / `CLIENT_PLAYER_NAME` | ダイアログ初期値 | GameServer / room1 / Player |

優先度: コマンドライン引数 > OSの環境変数 > `.env` > コード内の既定値。

## 起動方法

### クライアント (GUI)

```bash
# ゲームランチャー (メニューからゲームを選択)
python3 client/main.py

# 各ゲームの単体起動も可能
python3 client/blackjack.py
```

### 専用サーバ (Docker Compose 推奨)

```bash
cp sample.env .env          # 初回のみ。ポートや TUNNEL_TOKEN を編集
docker compose up -d --build
docker compose logs -f      # ログ確認
docker compose down         # 停止
```

ゲームサーバ (`game-server`) と **Cloudflare Tunnel** (`cloudflared`) が
起動する。トンネルを使わない場合は `docker-compose.yml` の
`cloudflared` サービスをコメントアウトする。

### 専用サーバ (直接起動)

```bash
# 設定は .env から取得。引数で上書きも可能
python3 server/server.py [--port 80] [--name GameServer] [--bind 0.0.0.0]
```

## 接続モード

ゲーム起動時の接続設定ダイアログで3つのモードを選べる。
アドレス欄は **IPアドレス / ドメイン名 / URL (`http://`・`https://`)** の
いずれでも指定できる。URLを指定した場合はポート番号欄は無視される
(リバースプロキシやhttps経由の接続用)。

| モード | 説明 | ポート |
|---|---|---|
| ホストモード | 自分のPCが全IP (0.0.0.0) の指定ポートでHTTP待受し、自身もプレイヤーとして参加 | ゲーム別 (35555〜) |
| クライアントモード | ホストのアドレスとポートを指定して参加 | ホストと同じ |
| サーバー接続モード | 専用サーバ (`server.py`) に **サーバ名 + ルームID** を指定して参加 | 80 (既定) |

### 大学・会社のネットワークからの接続

外向きの通信が 80/443 番しか通らないネットワークからでも遊べるように、
通信はすべてHTTPで行う。

- **Cloudflare Tunnel (推奨)**: サーバ側のポート開放・固定IPが一切不要。
  docker compose に同梱の `cloudflared` が外向きにトンネルを張り、
  クライアントは `https://<公開ホスト名>` (443/https) で接続できる。
  手順は下記「Cloudflare Tunnel の設定」を参照
- 専用サーバを **80番ポート** で直接公開する場合は、アドレス欄に
  ドメイン名かIPを書くだけで接続できる (ポート欄は既定の80のまま)
- 自前で **443/https** 公開する場合は、nginx / Caddy 等のリバースプロキシで
  TLSを終端して `server.py` (任意のポート) へ転送し、クライアントは
  アドレス欄に `https://game.example.com` のようにURLを指定する
- クライアントは **OSのプロキシ設定を自動で利用**する
  (大学のHTTPプロキシ経由でも接続可)

### Cloudflare Tunnel の設定

1. [Cloudflare Zero Trust](https://one.dash.cloudflare.com/) →
   Networks → Tunnels でトンネルを作成し、**トークン**をコピーする
2. `.env` の `TUNNEL_TOKEN=` に貼り付ける
3. トンネルの **Public Hostname** を追加し、Service に
   `http://game-server:80` を指定する (`SERVER_PORT` を変えた場合は同じ番号)
4. `docker compose up -d --build` で起動
5. クライアントは接続設定のアドレス欄に `https://<公開ホスト名>` を入力
   (URL指定時はポート欄は無視される)

### ルーム機能 (専用サーバ)

- 1つのサーバが1ポートで全ゲームのルームを同時にホストする
- ルームは最初の参加者が入ると自動作成され (そのゲーム種類で確定)、全員退出で自動削除
- サーバ名が一致しない接続、既存ルームへの別ゲームでの参加は拒否される
- 最初の参加者が**ルームマスター**としてゲーム開始権限を持つ (退出時は自動引き継ぎ)
- ルーレット等の制限時間の管理 (残り時間配信・時間切れの自動抽選) もサーバが行う

## 収録ゲーム

進行方式は3系統。**ターン制で1人を長時間待つ設計は排除**している。

| 方式 | 説明 | ゲーム |
|---|---|---|
| 時限ベット型 | 全員が同時にベット。「全員が賭け終了」または制限時間で締切 → 抽選演出 → 一括精算 | ルーレット / バカラ / クラップス / シックボー / マネーホイール |
| 一斉選択型 | 全員が同時に行動を選択し、揃った瞬間に一括で1ターン進む | ブラックジャック |
| 自由プレイ型 | 各自が好きなタイミングでプレイ (順番の概念なし) | スロット |
| ターン制 | 手番が回る (ポーカー/カードゲームの性質上) | テキサスホールデム / Color & Numbers |

## アーキテクチャ

### 通信の仕組み

- **権威ホスト方式**: ゲーム状態はホスト (GUIホストまたは専用サーバ) のエンジンだけが持ち、
  クライアントはアクションを送信して状態を受け取り描画するだけ。チート・不整合が起きにくい
- **プロトコル**: HTTP/1.1 + JSON。クライアント側は `client/httpnet.py`
  (GUIホスト用の最小HTTPサーバ + QNetworkAccessManager 製クライアント)、
  専用サーバ側は `server/server.py` (標準ライブラリ `http.server` ベース、
  スレッド + ロングポーリング)。エンドポイントは両者共通

  | エンドポイント | 内容 |
  |---|---|
  | `POST /api/join` | `hello` メッセージで参加 → `{"sid": セッションID}` |
  | `POST /api/send?sid=` | ゲームメッセージ (`action` / `start`) の送信 |
  | `GET /api/poll?sid=` | 受信キューの取得。**ロングポーリング** (最長20秒保持) で低遅延に配信 |
  | `POST /api/leave?sid=` | 明示的な退出 |
  | `GET /api/ping` | 死活確認 |

  poll が45秒間途絶えたセッションは退出扱いになる。
- **メッセージJSON** (HTTPのボディとして従来と同じ構造を運ぶ):
  - クライアント → ホスト: `hello` (参加。サーバ接続時は `server`/`room`/`game` を含む) / `action` / `start`
  - ホスト → クライアント: `welcome` (ID割当) / `state` (全状態) / `info` (ログ) / `error`
- **視点別の状態配信**: `engine.public_state(viewer)` がプレイヤーごとの視点で状態を生成。
  ホールデムの手札・Color & Numbersの手札などの秘匿情報は本人にしか送信されない

### 演出とネタバレ防止

抽選演出 (ルーレットのスピン、バカラのカード公開、ダイスロール、
ブラックジャックのディーラー公開等) は各クライアントのローカル処理。
演出中は結果メッセージ・勝敗・精算後のチップ額をマスクした状態で描画し、
演出が終わった瞬間に公開する。通信プロトコルには影響しない。

演出の設計方針は「短く・入力をブロックせず・毎回の再描画で再発火しない」:

- **フェードイン** (`common.fade_in`): 新しく現れた要素だけを 0.2〜0.4 秒で
  フェード表示。前回描画したカード列と比較し、増えた/変わった札のみ対象に
  することで、状態の再配信ごとに画面がチラつくことを防ぐ
- **結果の一括公開**: 抽選演出の終了と同時に、各プレイヤーの WIN/LOSE と
  勝敗メッセージがフェードインで登場する
- **締切の予告**: 時限ベット型ゲームは残り10秒を切ると残り時間表示が
  赤に変わる (時間切れで慌てないための警告)
- **進行ロック**: 演出中は「次のラウンド」ボタンを無効化し、
  演出のスキップ・置き去りが起きないようにしている

### ファイル構成

```
docker-compose.yml   サーバのコンテナ起動定義 (.env を読み込む)
sample.env           設定サンプル (cp sample.env .env して使う)

shared/              ★ クライアント/サーバ共有 (純Python・Qt非依存)
  cards.py             トランプ定義
  envfile.py           .env ローダ
  engines/             全ゲームのエンジン (状態の権威・ルール・精算)
    betting_base.py      時限ベット型エンジン基底
    blackjack.py / holdem.py / roulette.py / baccarat.py / craps.py
    sicbo.py / money_wheel.py / slots.py / color_numbers.py

client/              ★ GUIクライアント (PySide6)
  main.py              ランチャー (メニュー画面)
  common.py            接続ダイアログ / カード描画 / 演出ヘルパー (fade_in) /
                       ゲームウィンドウ基底クラス (GameWindowBase)
  httpnet.py           HTTP通信 (GUIホスト用サーバ + クライアント)
  betting.py           時限ベット型ウィンドウ基底 / ダイス演出ウィジェット
  blackjack.py ほか    各ゲームのウィンドウ (エンジンは shared から import)
  requirements.txt     PySide6

server/              ★ ヘッドレス専用サーバ (標準ライブラリのみ)
  server.py            全ゲーム・ルーム機能・ロングポーリング
  Dockerfile           python:3.12-slim ベース (pip install 不要)
```

### 新しいゲームの追加方法

1. エンジン (ロジック) を `shared/engines/` に書く。時限ベット型なら
   `betting_base.BettingEngineBase` を継承し `resolve()` (抽選と精算) と
   `extra_state()` を実装するだけ
2. ウィンドウは `client/` に書く。`betting.BettingWindowBase`
   (または `common.GameWindowBase`) を継承し、盤面の構築・描画・演出フックを実装
3. `client/main.py` の `GAMES` と `server/server.py` の `GAMES` に登録

## サーバ運用メモ

- サーバは**標準ライブラリのみ**で動く (PySide6不要)。Dockerイメージは
  `python:3.12-slim` ベースで `pip install` なし
- デフォルトの待受は **80番ポート** (HTTP標準)。Dockerならそのまま公開できる。
  直接起動でLinuxの1024未満ポートを使うには root 権限か
  `CAP_NET_BIND_SERVICE` が必要 (macOSはそのまま可)。
  権限を避けたい場合は `.env` で `SERVER_PORT=8080` 等にし、
  リバースプロキシで 80/443 から転送する
- **https (443) で公開する例** (nginx):

  ```nginx
  server {
      listen 443 ssl;
      server_name game.example.com;
      # ssl_certificate ...; ssl_certificate_key ...;
      location / {
          proxy_pass http://127.0.0.1:8080;   # SERVER_PORT=8080 のコンテナへ
          proxy_read_timeout 60s;             # ロングポーリング (最長20秒) 用
      }
  }
  ```

  クライアントはアドレス欄に `https://game.example.com` を指定する
- `--bind` (または `SERVER_BIND`) にドメイン名を指定すると `getaddrinfo` で
  解決してバインドする (Docker利用時はコンテナ内で常に 0.0.0.0)
- `GET /api/ping` で死活確認できる (`{"ok": true, "app": "casino-games"}`)
- ログは標準出力 (タイムスタンプ付き)。直接起動は Ctrl+C、
  Docker は `docker compose down` で終了
- macOSでは初回起動時にファイアウォールの受信許可が必要な場合がある
